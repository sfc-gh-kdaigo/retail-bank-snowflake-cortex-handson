# Semantic Viewにおける日本語カラム名の懸念事項

本ドキュメントは、Semantic View（Cortex Analyst用）で日本語カラム名を使用する際の懸念事項と推奨事項をまとめたものです。

---

## 結論

**技術的には動作するが、精度向上のためにベストプラクティスに従うことを推奨**

---

## 動作面での確認

| 項目 | 状況 |
|:---|:---|
| 日本語カラム名のSQL実行 | ✅ ダブルクォートで囲めば動作する |
| Semantic View定義 | ✅ 日本語カラム名を含むテーブルを参照可能 |
| Cortex Analyst | ✅ LLM（Claude等）は日本語を理解可能 |

---

## 精度に影響する可能性のある懸念事項

### 1. LLMのSQL生成時のエスケープ漏れリスク

Cortex AnalystはLLMを使ってSQLを生成します。日本語カラム名の場合、LLMが以下のようなミスをする可能性があります。

```sql
-- 正しい（期待する出力）
SELECT "顧客番号", "漢字氏名１" FROM "顧客基本属性情報＿月次"

-- LLMが誤って生成する可能性
SELECT 顧客番号, 漢字氏名１ FROM 顧客基本属性情報＿月次  -- クォートなし = エラー
```

### 2. 全角文字の解釈

ファイル名・カラム名に含まれる全角文字（特に `＿` 全角アンダースコア）は注意が必要です。

- `顧客基本属性情報＿月次` ← 全角アンダースコア使用
- 文字化けやエンコーディングの問題が発生する可能性
- コピー＆ペースト時に意図せず半角に変換されるリスク

### 3. Synonyms設定の重要性

Cortex Analystの精度は **Synonyms（同義語）** の設定に大きく依存します。日本語カラム名の場合、ユーザーが様々な言い方で質問する可能性があるため、十分なSynonymsが必要です。

例：
- 「顧客名」「氏名」「名前」→ `漢字氏名１`
- 「顧客区分」「個人法人区分」→ `人格コード`

### 4. LLMのトークン消費

日本語は英語と比較してトークン効率が低いため、同じ情報量でもより多くのトークンを消費します。これは以下に影響する可能性があります。

- レスポンス時間の増加
- コンテキストウィンドウの消費量増加
- 複雑なクエリでの精度低下リスク

## 日本語カラム名 vs 英語カラム名の比較

| 項目 | 日本語カラム名 | 英語カラム名 |
|:---|:---|:---|
| SQL生成精度 | △ やや不安定 | ◎ 安定 |
| エスケープ問題 | △ 発生リスクあり | ◎ なし |
| ユーザー体験 | ◎ 自然な日本語 | ○ Synonymsで対応可 |
| メンテナンス | △ 特殊文字注意 | ◎ 容易 |
| デバッグ | △ ログの可読性低下 | ◎ 容易 |
| 国際化対応 | × 困難 | ◎ 容易 |

## 推奨事項

### 方法A: 日本語カラム名を使用する場合

以下の対策を実施することで、精度への悪影響を最小限に抑えることができます。

#### 1. Synonymsを充実させる

```yaml
# Semantic View設定例
columns:
  - name: "漢字氏名１"
    synonyms:
      - 顧客名
      - 氏名
      - 名前
      - お客様名
  - name: "人格コード"
    synonyms:
      - 顧客区分
      - 個人法人区分
      - 顧客タイプ
```

#### 2. Descriptionを詳細に設定する

各カラムの意味、値の範囲、使用例を記載し、LLMがカラムの意図を正確に理解できるようにします。

```yaml
columns:
  - name: "人格コード"
    description: |
      顧客の人格を識別するコード。
      1: 個人、2: 法人、3: 個人事業主。
      顧客区分の集計や個人/法人の分析に使用。
```

#### 3. サンプルクエリを多く登録する

様々なパターンのクエリを登録することで、LLMの学習精度が向上します。

- 最低でも各テーブルをカバーするクエリを用意
- 複数テーブルのJOINを含むクエリも含める
- ビジネスKPIの計算パターンを網羅

#### 4. メトリクス定義を活用する

複雑な計算式はメトリクスとして事前定義し、LLMが正確なSQLを生成できるようにします。

```yaml
metrics:
  - name: NET_FUND_INFLOW
    synonyms:
      - 純資金流入額
      - 資金流入
    expression: |
      SUM(CASE WHEN "入払区分" = 1 THEN "取引金額" ELSE 0 END) -
      SUM(CASE WHEN "入払区分" = 2 THEN "取引金額" ELSE 0 END)
```

### 方法B: 英語カラム名版を使用する（推奨）

より高い精度と安定性を求める場合は、英語カラム名版スキーマ（`RETAIL_BANKING_EN`）の使用を推奨します。

```sql
-- 英語版スキーマを使用
USE SCHEMA RETAIL_BANKING_DB.RETAIL_BANKING_EN;
```

日本語対応はSynonymsで十分に実現できます。

```yaml
tables:
  - name: CUSTOMER_ATTRIBUTES_MONTHLY
    synonyms:
      - 顧客基本属性情報
      - 顧客マスタ
      - 顧客情報
columns:
  - name: CUSTOMER_NAME
    synonyms:
      - 顧客名
      - 氏名
      - 名前
```

## 検証方法

実際に動作させて精度を確認することを推奨します。

### 検証ステップ

1. Semantic Viewを作成
2. 以下のパターンで質問を投げてSQLの正確性を確認
   - シンプルな集計（COUNT, SUM）
   - フィルタ条件付き（WHERE句）
   - 複数テーブルのJOIN
   - メトリクス計算（純資金流入額など）
3. エラーが発生した場合はSynonyms/Description/サンプルクエリを追加
4. 再検証

### 確認ポイント

- [ ] 日本語カラム名が正しくクォートされているか
- [ ] 全角文字が文字化けしていないか
- [ ] JOINの結合条件が正しいか
- [ ] メトリクスの計算式が正確か
